# 用户故事模板（优化版）

## 标题
- 简短明确，含业务意图（最多 8-10 个词）

## 一句话概述
- 用一句话描述目标和价值（谁 / 想要 / 为了）

## 用户故事（标准格式）
- 作为 <角色>，我想要 <功能/动作>，以便 <价值/收益>

## 背景与动机
- 业务背景、触发场景、现状痛点

## 需求详情
- 功能拆分（要点形式）
- 关键流程步骤（用序号列出用户主要路径）

## 验收标准（AC） — 必须可测试、明确
- 列表式验收项，使用 GIVEN / WHEN / THEN 格式：
  - GIVEN <前置条件>
    WHEN <操作>
    THEN <预期结果>

## 非功能需求
- 性能（响应时间、并发）、安全、隐私、可用性、可访问性、兼容性、本地化

## 设计 / 原型
- 相关链接或截图（指向 Figma/Mockup）

## 数据与指标（可选但推荐）
- 成功衡量：KPI / 指标（例如转化率 +2% / 错误率 < 0.1%）

## 风险与假设
- 主要风险、未验证的假设、降级方案

## 依赖与约束
- 外部系统、第三方、其他团队或里程碑依赖

## 测试用例（示例）
- 简要列出必需的测试场景（正例、负例、边界）

## 验收定义（Definition of Done）
- 代码完成并覆盖单元/集成测试
- UI/交互通过审查并有设计签字
- 自动化测试或手工回归测试通过
- 文档/使用说明更新
- 上线监控与回滚方案已准备

## 估时与优先级
- 估时（Story Points 或 人天）
- 优先级（P0/P1/P2 或 高/中/低）

## 备注 / 附加信息
- 相关讨论链接、参考资料、法律/合规要求

---

## 模板使用指南（快速检查清单）
- [ ] 是否符合 INVEST：独立（Independent）、可协商（Negotiable）、有价值（Valuable）、可估计（Estimable）、小（Small）、可测试（Testable）
- [ ] 是否包含清晰验收标准（GIVEN/WHEN/THEN）
- [ ] 是否列出依赖和风险
- [ ] 是否定义了衡量成功的指标
